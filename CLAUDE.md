## Project Overview

AeroSense is an aviation weather intelligence app built for a WindBorne Systems internship challenge. It shows live ASOS station weather data on an interactive Mapbox map, overlays animated wind particles, displays nearby flights via ADS-B, and runs AI-powered weather analysis via Gemini.

## Commands

### Frontend (React + Vite)
```powershell
npm run dev          # Start Vite dev server (port 5173)
npm run build        # Production build ‚Üí dist/
npm run lint         # ESLint check
npm run preview      # Preview production build locally
```

### Backend (Express)
```powershell
cd server
npm run dev          # nodemon app.js (port 3000, hot reload)
npm start            # node app.js (production)

```

Run both concurrently in two terminals. Vite proxies all `/api/*` requests to `localhost:3000` (configured in `vite.config.js`).

```powershell
npm run vercel-build  # Runs: npm install && cd server && npm install && cd .. && npm run build
```

## Architecture

### Two-process split, one serverless entrypoint

**Development**: Two separate processes ‚Äî Vite (frontend) and Express (`server/app.js`). Vite proxies `/api` to Express.

**Production (Vercel)**: A single serverless function at `api/index.js` handles all `/api/*` routes. The Express app from `server/` is imported directly into this function ‚Äî `server/app.js` detects `VERCEL=1` and skips `app.listen()`. Frontend is served as static files from `dist/`.

The `vercel.json` rewrite rule `"/api/(.*)" ‚Üí "/api"` routes all API traffic to `api/index.js`.

### Frontend component tree (`src/`)

```
App.jsx              ‚Üê all top-level state, data fetching, routing between Landing/Map views
‚îú‚îÄ‚îÄ Landing.jsx      ‚Üê first-visit splash screen (localStorage: 'aerosense-visited')
‚îú‚îÄ‚îÄ Controls.jsx     ‚Üê search bar, time slider, wind pause, compare mode, demo toggle
‚îú‚îÄ‚îÄ AeroMap          ‚Üê Mapbox GL map (react-map-gl), station markers, flight icons
‚îú‚îÄ‚îÄ WindOverlay      ‚Üê Canvas-based particle animation driven by wind_x/wind_y vectors
‚îú‚îÄ‚îÄ FlightOverlay    ‚Üê Headless component; fetches ADS-B flights, exposes refresh() via ref
‚îú‚îÄ‚îÄ Sidebar.jsx      ‚Üê Weather details, historical chart, flight list for selected station
‚îú‚îÄ‚îÄ FlightInfoPanel  ‚Üê Floating popup when a flight is clicked
```

**Key state pattern**: All data lives in `App.jsx`. Child components receive data and callbacks as props. `FlightOverlay` is the only exception ‚Äî it uses `forwardRef` + `useImperativeHandle` to expose a `refresh()` method to its parent.

### Backend service layers (`server/src/`)

```
routes/api.js             ‚Üê 4 routes: /stations, /weather, /flights, /analyze
controllers/              ‚Üê thin layer: validates params, calls services, returns JSON
services/
  windborneService.js     ‚Üê primary weather source (WindBorne ASOS API at sfc.windbornesystems.com)
  aviationWeatherService  ‚Üê METAR data source (aviationweather.gov)
  unifiedWeatherService   ‚Üê merges METAR + WindBorne; prefers METAR (<30 min) then WindBorne
  openskyService.js       ‚Üê ADS-B Exchange (adsb.lol) for flight positions
  geminiService.js        ‚Üê Gemini 2.5 Flash; analyzes weather history; 1-hour cache
utils/
  cache.js                ‚Üê Upstash Redis in prod; in-memory Map fallback in dev
  stationMapper.js        ‚Üê maps WindBorne station IDs to ICAO codes for METAR lookup
  validation.js           ‚Üê Zod schemas for WindBorne API responses
```

### Data flow for weather

1. `GET /api/weather?station={id}` ‚Üí `weatherController`
2. `unifiedWeatherService.getUnifiedWeather()` runs METAR + WindBorne in parallel
3. METAR takes priority if data age < 30 min; WindBorne fills historical `points[]`
4. Response shape: `{ points: [...], current: {...}, source, dataAge, isRealTime, degraded, degradedReasons }`
5. `App.jsx` stores this as `weatherData`; `timeIndex` selects which point to display
6. `WindOverlay` reads `wind_x`/`wind_y` from the selected point and animates particles

### Wind vector convention

`wind_x` = east component, `wind_y` = north component, both in knots. METAR uses meteorological FROM direction ‚Äî `aviationWeatherService` converts: `direction (FROM) ‚Üí radians = ((direction - 90) √ó œÄ) / 180`. In `WindOverlay`, `p.y -= wy` (canvas y-axis is inverted).

### Demo / outage fallback

When the WindBorne API is down:
- `stationController` returns `{ demoEligible: true }` instead of a stations array
- `App.jsx` detects this and shows a "Demo Mode" toggle
- Demo data is synthetic ‚Äî 36 hourly sinusoidal points generated by `windborneService.getDemoWeather()`
- Query param `?demo=true` on `/api/weather` triggers demo data

### Caching strategy

| Data | TTL | Stale fallback TTL |
|------|-----|--------------------|
| Stations list | 24 hours | 7 days |
| Weather history | 5 minutes | ‚Äî |
| Gemini analysis | 1 hour | ‚Äî |

Redis is required in production; missing credentials fall back to in-memory `Map` (unsuitable for serverless multi-instance).

## Environment Variables

**Required** (server):
- `UPSTASH_REDIS_REST_URL`
- `UPSTASH_REDIS_REST_TOKEN`

**Optional** (server):
- `GEMINI_API_KEY` ‚Äî enables `/api/analyze` endpoint
- `OPENSKY_CLIENT_ID` / `OPENSKY_CLIENT_SECRET` ‚Äî authenticated OpenSky access (higher rate limits); currently unused since ADS-B Exchange is the primary flight source

Create `server/.env` for local development.

## Key Conventions

- Wind risk thresholds: `< 15 kt = SAFE`, `15‚Äì25 kt = CAUTION`, `> 25 kt = DANGER` ‚Äî defined in `src/utils/aviation.js` and mirrored in the Gemini prompt. Do not change one without the other.
- Station normalization: WindBorne uses `station_id/station_name/latitude/longitude`; the frontend normalizes these to `id/name/lat/lng` in `App.jsx:onStationSelect`. Always normalize before storing in state.
- `src/lib/utils.js` contains the `cn()` Tailwind class merger (clsx + twMerge pattern).
